Ниже представлен **полный технический справочник** по 4G-роутеру (CPE) на базе Android 4.4, составленный исключительно на основе данных, предоставленных в пяти источниках (нейронках). Всё, что здесь написано, взято из этих материалов без каких-либо домыслов и добавлений.

# Полный технический справочник 4G CPE-роутера на Android 4.4

## 1. Общее описание и архитектура

### 1.1. Что это за устройство
- **4G/LTE CPE-роутер** (Customer Premises Equipment) – принимает интернет через сотовую сеть (SIM-карту) и раздаёт его по Wi-Fi и Ethernet.
- Работает под управлением **Android 4.4.4 (API 19)**. Это полноценное Android-устройство, выполняющее функцию роутера.
- **Чипсет:** Qualcomm (стандарт для многих китайских USB-свистков/CPE).
- Основной процесс: `com.wowi.nanowebyl` (пакет APK `cpeweb.apk`).

### 1.2. Архитектурный стек
- **ОС:** Android 4.4.4.
- **Встроенный веб-сервер:** реализован в APK (вероятно, на базе NanoHTTPD или аналогичной Java-библиотеки). Слушает порт **8000**. Обрабатывает как статические файлы интерфейса, так и API-запросы.
- **Дополнительный CGI-сервер:** запущен пользователем через `busybox httpd` на порту **8080** (расширяет функциональность, позволяет отправлять AT-команды и выполнять длительные операции).
- **Прямой доступ к модему:** через устройство `/dev/smd11` (принимает AT-команды).
- **Доступ:** полный **ROOT** через ADB (после включения ADB).

### 1.3. Основные способы управления
1. **Основной API** – JSON RPC на порту 8000 (`/api/json`).
2. **CGI-скрипты** – на порту 8080 (`/cgi-bin/api`, `/cgi-bin/scan.cgi` и др.).
3. **Скрытые URI** встроенного сервера (`/proc_get`, `/elink_adb`, `/master_clear` и пр.).
4. **Прямая работа с `/dev/smd11`** (например, через shell).

## 2. Основной API (JSON RPC) – порт 8000

### 2.1. Общие характеристики
- **URL:** `http://<IP_роутера>:8000/api/json` (или просто `http://<IP>/api/json`, если сервер на стандартном порту 80, но в материалах указано, что Java-сервер слушает 8000, а фронтенд обращается через прокси).
- **Метод:** POST.
- **Content-Type:** `application/json`.
- **Аутентификация:**
  - После успешного логина сервер возвращает `sessionId`.
  - Во всех последующих запросах `sessionId` передаётся **двумя способами**:
    - В теле запроса как поле `sessionId`.
    - В HTTP-заголовке `Authorization: <sessionId>`.
- **Формат ответа:** JSON-объект с обязательным полем `reply`. При успехе также может присутствовать поле `fields` (данные) или `session` (при логине).
- **Таймауты запросов:**
  - Обычные запросы – 10 секунд.
  - `setGW`, `setWifi` – 20 секунд (из-за перезагрузки).
  - `login` – 2 секунды.

### 2.2. Структура запроса
```json
{
  "fid": "ИМЯ_ФУНКЦИИ",
  "fields": {
    "ПАРАМЕТР_1": "ЗНАЧЕНИЕ",
    "ПАРАМЕТР_2": "ЗНАЧЕНИЕ"
  },
  "sessionId": "..."  // для авторизованных запросов
}
```

### 2.3. Коды ответов (поле `reply`)
| Значение | Описание |
|----------|----------|
| `"ok"` | Запрос выполнен успешно |
| `"Authorization"` | Сессия недействительна или истекла – требуется повторный вход |
| `"empty"` | Не заполнено обязательное поле (например, пароль при логине) |
| `"password_error"` | Неверный пароль (при логине, смене пароля, переключении SIM) |
| `"net_error"` | Ошибка сети (возможно, сервер недоступен) |
| `"checksum"` или `"checksum_error"` | Ошибка контрольной суммы при обновлении прошивки |
| `"error"` (иногда с `message`) | Прочие ошибки |

### 2.4. Время жизни сессии
- Сессия живёт **5 минут** бездействия (300000 мс). После этого фронтенд показывает диалог и принудительно выходит.

### 2.5. Валидация на стороне клиента
- **SSID:** длина 5–32 символа.
- **Пароль Wi-Fi:** если не открытая сеть, длина 8–63 символа (печатные ASCII).
- **Пароль администратора:** длина 3–20 символов.

## 3. Полный список команд (FID)

Ниже приведены все обнаруженные в материалах идентификаторы функций (FID) с описанием и необходимыми полями.

### 3.1. Аутентификация и сессия

| FID | Описание | Поля запроса | Ответ / Примечания |
|-----|----------|--------------|---------------------|
| `init` | Инициализация приложения, получение начальных настроек (например, языка) | `{}` | Возвращает `fields.language` (текущий язык: en, ru, es, zh) |
| `login` | Аутентификация пользователя | `username`, `password` | При успехе возвращает `session` (токен сессии) |
| `changePassword` | Смена пароля администратора | `oldPassword`, `newPassword` | `reply: "ok"` |
| `logout` | Завершение сессии | `{}` | `reply: "ok"` |

### 3.2. Чтение/запись данных

| FID | Описание | Поля запроса | Примечания |
|-----|----------|--------------|------------|
| `queryFields` | Запрос текущих значений одного или нескольких полей | `fields`: объект с именами полей (значения игнорируются) | Возвращает `fields` с актуальными значениями |
| `setFields` | Универсальная запись значений полей | `fields`: объект с новыми значениями | Используется для изменения IMEI, режимов USB, сети, TTL, частотных диапазонов и т.д. |

### 3.3. Настройки Wi-Fi и LAN

| FID | Описание | Поля запроса | Примечания |
|-----|----------|--------------|------------|
| `setWifi` | Настройка основной точки доступа Wi-Fi | `wifiApSwitch` ("on"/"off"), `ssidName`, `ssidBroadcast` (bool), `ssidSecureMode` ("NONE"/"WPA_PSK"/"WPA2_PSK"), `ssidPassword`, `ssidMaxUserCount` (число) | |
| `setGW` | Настройка локальной сети (Gateway) | `ipAddress`, `subnetMask`, `dhcpSwitch` ("on"/"off"), `dhcpFrom`, `dhcpTo`, `dhcpLeases` (часы), `ethType` ("lan"/"wan") | `ethType` – тип Ethernet-порта (чтение/запись) |

### 3.4. APN (точки доступа)

| FID | Описание | Поля запроса | Примечания |
|-----|----------|--------------|------------|
| `queryApn` | Запрос текущих настроек APN и списка доступных конфигураций | `{}` или `selectId` | Возвращает `currentApn`, `apnMode`, `currentConfig`, `apnConfigs` (массив `{id, name}`) и другие поля конфигурации |
| `setApn` | Создание новой или изменение существующей APN-конфигурации | `id` (число, при редактировании), `selectId`, `configName`, `pdpType` ("IPv4"/"IPv6"/"IPv4V6"/"IP"), `apn`, `mcc`, `mnc`, `authtype` (-1...3), `apnUser`, `apnPassword`, `apnProxy`, `apnPort` | |
| `deleteApn` | Удаление существующей APN-конфигурации | `selectId` (число) | |
| `setDefaultApn` | Установка конфигурации APN по умолчанию | `selectId` (число) | |

### 3.5. SIM-карты

| FID | Описание | Поля запроса | Ответ / Примечания |
|-----|----------|--------------|---------------------|
| `switchSimCard` | Переключение между физическими SIM-картами (если поддерживается) | `simCardCurrent` (индекс новой SIM: 0,1...), `password` (если требуется) | Возвращает `reply: "ok"`, `"empty"` (нет пароля) или `"password_error"` |

### 3.6. Системные операции

| FID | Описание | Поля запроса | Примечания |
|-----|----------|--------------|------------|
| `rebootSystem` | Перезагрузка устройства | `{}` | |
| `factoryReset` | Сброс до заводских настроек (с последующей перезагрузкой) | `{}` | |
| `upgradeWebServer` | Запуск процесса обновления веб-интерфейса (после загрузки APK через `/upload/webserver`) | `{}` | |
| `upgradeSystem` | Запуск обновления прошивки (после загрузки файла через `/upload/system` и опционально MD5) | `{}` | При ошибке контрольной суммы возвращает `reply: "checksum"` |

### 3.7. Диагностика (Watchdog / Ping)

| FID | Описание | Поля запроса |
|-----|----------|--------------|
| `testConnect` | Включение/выключение проверки связи | bool |
| `pingAddress1`, `pingAddress2`, `pingAddress3` | IP-адреса или домены для пинга | строки |
| `testInterval` | Интервал между проверками (секунды) | число |
| `testTimes` | Количество неудач перед перезагрузкой | число |

*Примечание: в некоторых источниках эти поля могут устанавливаться через `setFields`, а не отдельным FID.*

### 3.8. Дополнительные FID, встречающиеся в некоторых версиях прошивки

| FID | Описание | Где встречено |
|-----|----------|----------------|
| `updateTtl` | Изменение TTL (возможно, отдельная команда) | В некоторых JS-файлах |
| `updateBand` | Выбор частотного диапазона LTE | В некоторых JS-файлах |
| `selectLanguage` | Сохранение выбранного языка на сервере | В версии с мультиязычностью |
| `USSDType` / `DeviceInbox` / `DeviceOutbox` / `Drafts` / `SimSms` / `SmsSettings` | Управление SMS и USSD | В прошивках с поддержкой SMS |

## 5. Особые поля и скрытые функции

### 5.1. Смена IMEI
- Для записи IMEI через `setFields` требуется дополнительное поле **`zxcvbn`** с паролем `"zxcvbn"` (хардкод в коде).
- Пример запроса:
  ```json
  {"fid":"setFields", "fields":{"imei":"123456789012345", "zxcvbn":"zxcvbn"}, "sessionId":"..."}
  ```
- После успешной записи отправляется broadcast `elink.action.limitSpeed` с extra `imei`.

### 5.2. Переключение SIM-карты
- Поле `simCardCurrent` используется как для чтения, так и для записи.
- При изменении отправляется broadcast `android.intent.action.ELINK_SIM_SWITCH` с extra `sim_slot`.

### 5.3. Скрытые URI на встроенном сервере (порт 8000)

| URI | Метод | Описание | Параметры | Примечания |
|-----|-------|----------|-----------|------------|
| `/proc_get` | GET | Включение/выключение ADB | `debug_enable=1/0`, `action=System_Qualcomm`, `cmd=Model` | Требует пароль `qhyl` (проверяется через свойство `persist.ufi.ft.simswitchpword`) |
| `/elink_adb` | GET | Управление ADB | `adb=1/0` | Устанавливает `adb_enabled` в `Settings.Global` |
| `/master_clear` | GET | Сброс к заводским | – | |
| `/config` | ? | Изменение параметров Wi-Fi (для WeChat-интеграции) | – | |
| `/summary`, `/deviceInfo` | GET | Возвращают JSON с информацией о точке доступа | – | |
| `/custom/logo.png` | GET | Кастомный логотип | – | путь из `Settings.Global` `YKM_KEY_WEBLOGO_PATH` |
| `/upload/webserver` | POST | Загрузка APK для обновления веб-интерфейса | – | требует `Authorization` |
| `/upload/system` | POST | Загрузка файла прошивки | – | требует `Authorization` |
| `/upload/system_md5` | POST | Загрузка MD5-файла для прошивки | – | требует `Authorization` |

## 6. Дополнительный CGI-сервер (порт 8080)

Запущен через `busybox httpd`, скрипты находятся в `/data/local/web/cgi-bin/`. Используется для расширения функциональности.

### 6.1. Общий CGI-скрипт `api`
- **Назначение:** прокси для AT-команд и запросов к Java-серверу.
- **Формат запроса (POST):**
  - Если в теле есть поле `at_cmd` (или `atcmd`), команда отправляется на `/dev/smd11`, ответ возвращается в JSON.
  - Иначе запрос проксируется на внутренний Java-сервер (порт 8000).
- **Пример запроса AT-команды:**
  ```json
  {"at_cmd": "AT+CSQ"}
  ```
- **Пример ответа:**
  ```json
  {"reply": "ok", "response": "+CSQ: 26,99\r\nOK\r\n"}
  ```
- **Ограничения:** для коротких команд таймаут 5 секунд, для длительных лучше использовать асинхронный `scan.cgi`.

### 6.2. Асинхронный скрипт `scan.cgi` для `AT+COPS=?`
- **GET-параметры:**
  - `action=start` – запускает фоновое сканирование.
  - `action=status` – проверяет статус.
- **Ответы:**
  - `{"status": "started"}` или `{"status": "error", "message": "..."}`
  - `{"status": "scanning"}` – ещё идёт.
  - `{"status": "done", "data": "..."}` – готово, `data` содержит сырой ответ модема.
  - `{"status": "idle"}` – сканирование не запущено.
- **Механизм:** использует блокировку через `/data/local/tmp/modem_cops.lock` и сохраняет результат в `/data/local/tmp/modem_cops.txt`.

## 7. Работа с последовательным портом `/dev/smd11`

### 7.1. Особенности
- Порт занят основным приложением (`com.wowi.nanowebyl`). При прямой записи ответы могут перехватываться или приходить вперемешку.
- Для успешного обмена необходимо **предварительно открыть порт на чтение** (запустить `cat` в фоне).
- После использования порт нужно освободить (убить все `cat`).

### 7.2. Пример ручного теста
```bash
cat /dev/smd11 > /tmp/out &
echo -e "AT+CSQ\r" > /dev/smd11
sleep 2
kill %1
cat /tmp/out
```

### 7.3. Права доступа
```bash
chmod 666 /dev/smd11
```

## 9. Что устройство НЕ УМЕЕТ (ограничения прошивки)

1. **Режим репитера (WISP / Wi-Fi как WAN) – не поддерживается.** Устройство может только раздавать Wi-Fi, но не может подключаться к чужой Wi-Fi сети как клиент. Нет API, драйверов или интерфейса для сканирования и подключения к внешним точкам доступа.
2. **Сканирование сетей операторов – нет встроенной команды.** В списке FID отсутствует команда типа `scanNetwork`. Нет соответствующего поля в `queryFields` или `setFields`. Для сканирования приходится использовать CGI-скрипты и прямые AT-команды.
3. **Расширенные возможности OpenWrt – отсутствуют.** Прошивка основана на Android и самописном Java-приложении, не имеет ничего общего с OpenWrt.
4. **Произвольные AT-команды – не принимаются через основной API.** Встроенный Java-сервер не поддерживает FID для AT-команд. Для этого используется CGI-интерфейс или прямой доступ к `/dev/smd11`.
5. **Некоторые поля только для чтения** – например, `imei` можно прочитать, но записать только через специальную форму с паролем `zxcvbn`.

## 11. Известные проблемы и препятствия

- **Хардкод в APK:** файлы веб-интерфейса (HTML/JS/CSS) зашиты внутри `/system/priv-app/cpeweb.apk` в папке `assets/`. Приложение не читает внешние файлы из `/data/data/.../files/www/`.
- **CORS Policy:** если запустить свой веб-сервер на другом порту (например, 8080), браузер заблокирует запросы к API на порт 8000 из-за политики CORS. Решение – прокси-сервер, добавляющий заголовок `Access-Control-Allow-Origin: *`.
- **Порт smd11:** занят основным приложением. При одновременном доступе возможны конфликты.
- **Сессия живёт 5 минут** бездействия – требуется периодический опрос или перелогин.

## 12. Что нужно для создания кастомного интерфейса (ТЗ из нейронки 1)

Варианты реализации:
1. **Inject-метод:** декомпилировать APK, заменить assets, собрать заново.
2. **Proxy-метод:** написать микро-прослойку (Python/Go/Bash) на модеме, которая добавляет заголовок `Access-Control-Allow-Origin: *` к ответам родного API.
3. **Standalone-метод:** полностью игнорировать родной `api/json`, написать свой API на Bash (через CGI), общающийся с `/dev/smd11` напрямую и отдающий JSON.

Устройство имеет root. Команда `am force-stop com.wowi.nanowebyl` убивает сервер. Команда `mount -o remount,rw /system` позволяет менять системные файлы.
